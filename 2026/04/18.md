### わかったこと
- 保存ボタンを押したときにIDとcontentを渡す必要があるのだが、IDとcontentを普通に書いただけではダメだった。
mapメソッドの中で回しているからこそ、そのメッセージのIDやコンテンツが使える状態なのであって、mapメソッドの外だったらもうそれらは使えないんだなというのを、今自分でコードを書いて実行してみて初めて気づいた。
    - mapメソッドの外だったら「今の状態」を渡すことになるから、useStateで設定している editingId と editingText で渡すのが正しいんだなということがコードを書いて分かった。こういう微妙なところは、やはり自分でコードを書かないと分からない。
- 引数が不要な場合はアロー関数で包まずにロジックでかいた関数名を渡すだけでいい。例えばキャンセルの操作などは既に関数ができていて、id等の引数を渡す必要はない
- スキップすべきかなどは「Yes」か「No」で答えられるので、そういう状態を関数名で表現するときは、実務的には is や should で始まる命名規則があります。
    - 例えば、shouldSkip や isComposing などです。この命名は「読む人への約束」になります。is や should で始まる関数は true または false を返すという慣習を守ることで、他の人が読んでもコードの動きを予測できるようになります。
なので、真偽値（true / false）を扱う二択の場合は、この関数の命名規則を意識した方がいいですね。
- useRefはuseStateのレンダリングしない版
- useRef等のフックはコンポーネントのトップレベルでしか呼べない
